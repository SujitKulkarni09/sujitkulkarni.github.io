SQL in three sentences

SQL is the language used to ask a database questions and get exact answers back — "show me," "count," "combine," "filter." A join is simply how you combine two related tables into one answer, based on a value they share. Everything else on this page is really just the details of exactly which rows survive each kind of question you can ask.

01 · First principles

Why SQL exists at all

Before relational databases, data lived in flat files or spreadsheets — every fact repeated everywhere it was needed, and every question meant writing custom code to dig through it by hand.

Flat files & spreadsheets

A customer's name and address get copied into every single order row. Update their address once, and you have to hunt down every copy, or the data quietly disagrees with itself.

vs

Relational databases

A customer's info lives in exactly one place. Every order just points to it. Update it once, and every order referencing it is instantly correct — that pointer is what a join follows.

This is normalization (stage 20 goes deeper): splitting data into separate, non-repeating tables, linked by shared keys. It solves the "update it in twelve places" problem — but it also means the data you actually want is now spread across multiple tables. SQL joins are the tool that puts it back together, exactly when you need it, without ever duplicating it at rest.

The single idea underneath this entire page: a join doesn't move or copy data. It's a temporary, computed view that exists only for the length of your query — the underlying tables stay exactly as normalized and duplication-free as before you asked.
02 · Orientation

Where SQL actually sits

Not a database itself. A language for talking to one — and one of the most durable pieces of technology in computing, older than almost everything else still in daily use.

03 · In practice

One tap on your phone, traced through SQL

Same story format as the other guides — this time, purely the database side. You open a food delivery app and tap "Order History."

📱

You tap "Order History." Your app sends one request: "give me this user's past orders."

📁

Behind that request, your orders live in one table, and the restaurant names they belong to live in a completely separate table — kept apart on purpose, so a restaurant's name is stored exactly once, not copied into every order row.

🔗

The app fires a LEFT JOIN connecting your orders to their restaurant names by a shared restaurant_id — pulling every one of your orders, even the one from a restaurant that closed down and no longer exists in the restaurants table.

📊

A GROUP BY quietly runs alongside it to show "12 orders from this restaurant" as a single summary line, instead of listing all twelve separately.

⚡

An index on your user ID is the only reason this search takes milliseconds instead of scanning every order ever placed by every user on the platform.

🔒

The whole read happens inside a lightweight transaction, guaranteeing you never see a half-updated order mid-write, even if someone else is placing a new order on the same table right now.

✅

Your order history appears. A join, a group, and an index just cooperated silently, in milliseconds, without you ever knowing their names — until right now.

Filtering & Sorting
Narrowing a table down to exactly what you need
▼
🔍
Think of it likeAsking the librarian not just for "mystery novels," but "mystery novels published after 2010, sorted newest first, just give me the top 5." Each extra clause narrows and shapes the answer further.
The order these clauses actually run in

SQL is written SELECT → FROM → WHERE → ORDER BY → LIMIT, but the database executes it almost backwards: it finds the table, filters rows, then picks and sorts columns. Understanding this order is what makes later stages like GROUP BY and window functions click.

terminalsql
SELECT name, signup_date
FROM customers
WHERE signup_date > '2024-01-01'
ORDER BY signup_date DESC
LIMIT 5;

04
Primary Keys & Foreign Keys
The exact mechanism a join follows
▼
🔑
Think of it likeEvery customer gets a unique membership number (primary key) at a library. Every book they check out just records that number (foreign key) instead of rewriting the customer's whole name and address onto every single book slip.
Why joins need a key to follow at all

A primary key uniquely identifies every row in a table — no two rows can share one. A foreign key is a column in one table that stores another table's primary key, creating a link between them. This link is the literal thing every JOIN ... ON clause is matching.

schema.sqlsql
CREATE TABLE orders (
  id INT PRIMARY KEY,
  customer_id INT REFERENCES customers(id), -- foreign key
  amount DECIMAL
);

This is exactly the relationship stage 05 onward will exploit: orders.customer_id points back to customers.id, and a join is simply the operation that follows that pointer to stitch the two tables together.

primary keyforeign keyreferential integrityone-to-many relationships
05
INNER JOIN
Only the rows that exist in both tables
▼
🤝
Think of it likeA guest list matched against who actually showed up. An INNER JOIN only keeps names present on both lists — invited-and-attended. Anyone invited who didn't come, or anyone who crashed the party uninvited, is dropped entirely.
Retrieves records with matching values in both tables

The overlapping middle of the Venn diagram. If a row in A has no matching row in B (or vice versa), it simply doesn't appear in the result at all — this is the strictest and most commonly used join.

terminalsql
SELECT *
FROM A
INNER JOIN B ON A.key = B.key;
A concrete version, with real table names
terminalsql
SELECT customers.name, orders.amount
FROM customers
INNER JOIN orders ON customers.id = orders.customer_id;
-- only customers who have placed at least one order
INNER JOINmatching rows onlyON clause
06
LEFT JOIN
Everything from the left table, matched where possible
▼
👤
Think of it likeA full class roster (left table), with each student's exam score filled in next to their name where one exists (right table) — and a blank left next to any student who simply never took the exam. Nobody gets dropped off the roster.
Retrieves all records from the left table and matching records from the right table

Every row from A appears in the result, whether or not it has a match in B. Where there's no match, B's columns come back as NULL.

terminalsql
SELECT *
FROM A
LEFT JOIN B ON A.key = B.key;
A concrete version
terminalsql
SELECT customers.name, orders.amount
FROM customers
LEFT JOIN orders ON customers.id = orders.customer_id;
-- every customer, with NULL where they've never ordered
LEFT JOINLEFT OUTER JOINNULL results
07
LEFT JOIN with NULL Check
Finding exactly what's missing on the right side
▼
🚫
Think of it likeTaking that same full class roster with blanks for missing exam scores, and then only keeping the rows where the blank actually exists — instantly producing "students who never took the exam," with zero manual scanning.
Filters only the records where there is no match in the right table (NULL values)

This is one of the most useful patterns in everyday SQL: a LEFT JOIN followed by WHERE B.key IS NULL becomes an "anti-join" — everything in A that has no counterpart in B at all.

terminalsql
SELECT *
FROM A
LEFT JOIN B ON A.key = B.key
WHERE B.key IS NULL;
The real question this answers
terminalsql
SELECT customers.name
FROM customers
LEFT JOIN orders ON customers.id = orders.customer_id
WHERE orders.id IS NULL;
-- "which customers have never placed a single order?"
anti-join patternWHERE ... IS NULLexclusive left
08
RIGHT JOIN
The mirror image of LEFT JOIN
▼
👥
Think of it likeThe same class roster idea, but flipped: start from the exam sign-in sheet (right table) instead, and match each name back to the class roster where possible — keeping every exam entry even if that name isn't on the official roster at all.
Retrieves all records from the right table and matching records from the left table

The exact mirror of LEFT JOIN. Every row from B appears, matched to A where possible, NULL where not. In practice, most engineers just rewrite a RIGHT JOIN as a LEFT JOIN with the table order swapped — they produce identical results.

terminalsql
SELECT *
FROM A
RIGHT JOIN B ON A.key = B.key;

-- identical result, rewritten as a LEFT JOIN:
SELECT *
FROM B
LEFT JOIN A ON A.key = B.key;
RIGHT JOINRIGHT OUTER JOINequivalent to a flipped LEFT JOIN
09
RIGHT JOIN with NULL Check
Finding exactly what's missing on the left side
▼
🚫
Think of it likeStarting from the exam sign-in sheet and only keeping the names that don't appear on the official class roster at all — instantly surfacing "who took this exam without being enrolled."
Filters only the records where there is no match in the left table (NULL values)

The mirror of stage 07's anti-join, now run from the right table's side.

terminalsql
SELECT *
FROM A
RIGHT JOIN B ON A.key = B.key
WHERE A.key IS NULL;
A real-world version of this question
terminalsql
SELECT orders.id
FROM customers
RIGHT JOIN orders ON customers.id = orders.customer_id
WHERE customers.id IS NULL;
-- "which orders reference a customer that no longer exists?" — an orphaned-record check
anti-join patternexclusive rightorphaned record detection
10
FULL OUTER JOIN
Everything from both tables, matched wherever possible
▼
🌐
Think of it likeMerging the class roster and the exam sign-in sheet into one combined list — every student who's on either list appears once, matched where their names align, with a blank wherever they only appeared on one side.
Retrieves all records when there is a match in either the left or right table

The complete union: everything from A, everything from B, matched where a key aligns, NULL filling in whichever side has no counterpart.

terminalsql
SELECT *
FROM A
FULL OUTER JOIN B ON A.key = B.key;

Not every database supports FULL OUTER JOIN natively (notably older MySQL versions) — it can always be simulated as a LEFT JOIN UNION a RIGHT JOIN when it isn't.

FULL OUTER JOINUNIONcomplete outer join
11
FULL JOIN with NULL Check
Everything that failed to match on either side
▼
⚖️
Think of it likeTaking that merged roster-and-sign-in-sheet list and keeping only the mismatches — students who never took the exam, and exam-takers who were never enrolled — the complete list of everyone who's on exactly one list and not the other.
Filters only the records where there is no match in either the left or right table (NULL values in either)

This is the "symmetric difference" from set theory: A and B minus their overlap, in one query.

terminalsql
SELECT *
FROM A
FULL OUTER JOIN B ON A.key = B.key
WHERE A.key IS NULL OR B.key IS NULL;

12
CROSS JOIN & SELF JOIN
Two joins that don't fit the A-meets-B pattern
▼
🔄
Think of it likeA CROSS JOIN is pairing every shirt with every pair of pants you own, to see all possible outfits — no matching condition at all, just every combination. A SELF JOIN is comparing a class roster against itself, to find, say, every pair of students who share the same birthday.
CROSS JOIN: every combination, no condition
terminalsql
SELECT shirts.name, pants.name
FROM shirts
CROSS JOIN pants;
-- 5 shirts x 4 pants = 20 rows, every combination
SELF JOIN: a table matched against itself
terminalsql
SELECT e1.name AS employee, e2.name AS manager
FROM employees e1
JOIN employees e2 ON e1.manager_id = e2.id;
-- every employee, paired with their manager, from one single table

The alias (e1, e2) is what makes a self join possible — SQL needs two distinct names to treat the same table as two separate copies within one query.

CROSS JOINSELF JOINtable aliasesCartesian product

Aggregate Functions & GROUP BY
Turning many rows into one summary number
▼
📊
Think of it likeInstead of listing every single receipt from every customer, handing back one line per customer: "total spent: $340." GROUP BY is what collapses many rows into that one summary line, per group.
The five aggregate functions used constantly
FunctionReturns
COUNT()How many rows
SUM()The total of a numeric column
AVG()The average of a numeric column
MIN() / MAX()The smallest / largest value
terminalsql
SELECT customer_id, COUNT(*) AS order_count, SUM(amount) AS total_spent
FROM orders
GROUP BY customer_id;
-- one row per customer, summarizing all of their orders

This is the step in query execution order (stage 03) that runs after WHERE filters rows, and before the results are sorted — which is exactly why filtering on an aggregate needs a different clause entirely (stage 14).

COUNT / SUM / AVGMIN / MAXGROUP BY
14
HAVING vs WHERE
Filtering rows, versus filtering the groups you just built
▼
⚖️
Think of it likeWHERE is a bouncer checking IDs before anyone enters the party (before grouping). HAVING is checking the total headcount of each group after they've already been sorted into rooms — "only keep rooms with more than 10 people in them."
Why you can't just use WHERE for both

A COUNT() or SUM() doesn't exist yet when WHERE runs — it's calculated during GROUP BY. HAVING exists specifically because you need a filter that runs after aggregation.

terminalsql
SELECT customer_id, COUNT(*) AS order_count
FROM orders
WHERE amount > 0                 -- filters individual rows, before grouping
GROUP BY customer_id
HAVING COUNT(*) > 5;              -- filters groups, after grouping
WHERE (pre-aggregation)HAVING (post-aggregation)query execution order
15
Subqueries
A query, used as the input to another query
▼
🔁
Think of it likeAnswering "who spent more than the average customer?" by first quietly calculating the average in your head, then comparing each person against that number — a subquery is exactly that inner calculation, done automatically inside the outer question.
A query nested inside another query
terminalsql
SELECT name
FROM customers
WHERE id IN (
  SELECT customer_id FROM orders WHERE amount > 1000
);
-- "customers who have placed at least one order over $1000"
When a subquery is the wrong tool

A subquery re-runs once per outer row in the worst case, and can be genuinely slow at scale. The same "customers with high orders" question is often better written as a JOIN with a DISTINCT — subqueries shine for readability and one-off checks, joins usually win on raw performance.

subqueriesIN / EXISTScorrelated subqueriessubquery vs join performance

CTEs & Recursive Queries
Naming a subquery so a complex question reads like plain English
▼
📜
Think of it likeWriting "let X = the average order value" at the top of a math problem, so every step afterward can just say "X" instead of re-writing the whole calculation each time. A CTE is exactly that — a named, reusable result set for one query.
A CTE (Common Table Expression), in practice
terminalsql
WITH high_spenders AS (
  SELECT customer_id, SUM(amount) AS total
  FROM orders
  GROUP BY customer_id
  HAVING SUM(amount) > 1000
)
SELECT customers.name, high_spenders.total
FROM customers
JOIN high_spenders ON customers.id = high_spenders.customer_id;
Where this becomes essential: recursive queries

A recursive CTE can reference itself, which is what makes it possible to query an org chart's full management hierarchy, or a category tree's every nested subcategory, in one single query — something a plain JOIN genuinely cannot do at an unknown depth.

WITH clauseCommon Table Expressionsrecursive CTEshierarchical data
17
Window Functions
Calculating across rows, without collapsing them like GROUP BY does
▼
🏆
Think of it likeGiving every runner in a race their individual finishing time and their rank, on the same results sheet — without hiding anyone or collapsing the list down to just "1st place." GROUP BY would collapse the race into one row per team; a window function keeps every runner visible.
The feature that separates intermediate SQL from advanced SQL

A window function performs a calculation across a set of related rows, but returns a value for every row, instead of collapsing them into one summary row like GROUP BY does.

terminalsql
SELECT name, amount,
  RANK() OVER (ORDER BY amount DESC) AS spend_rank
FROM orders;
-- every single order, now labeled with its rank, nothing collapsed
The pattern that shows up constantly in analytics work
terminalsql
SELECT customer_id, order_date,
  LAG(order_date) OVER (PARTITION BY customer_id ORDER BY order_date) AS prev_order
FROM orders;
-- each customer's previous order date, next to their current one — the basis of churn analysis
RANK / ROW_NUMBERLAG / LEADPARTITION BYOVER clause

Indexes & Query Performance
Why the same query can take 2 seconds, or 2 milliseconds
▼
📚
Think of it likeA book's index at the back. Without it, finding every mention of one word means reading the entire book page by page. With it, you jump straight to the right pages. An index costs a little extra space to maintain, in exchange for never scanning the whole book again.
What an index actually does

Without an index, the database performs a full table scan — checking every single row to find matches. An index maintains a sorted structure on one or more columns, letting the database jump almost directly to matching rows instead.

terminalsql
CREATE INDEX idx_customer_id ON orders(customer_id);
-- now filtering or joining on customer_id is dramatically faster
Reading the query planner's own explanation
terminalsql
EXPLAIN ANALYZE
SELECT * FROM orders WHERE customer_id = 42;
-- shows whether the database used the index, or fell back to a full scan

Indexes aren't free — every extra index slows down INSERT/UPDATE slightly, since the index itself has to be updated too. The real skill isn't "add indexes everywhere," it's knowing which columns are actually filtered or joined on often enough to justify one.

indexesfull table scanEXPLAIN / EXPLAIN ANALYZEquery planner

Transactions & ACID
Guaranteeing an operation never half-completes
▼
💳
Think of it likeA bank transfer between two accounts. Money must leave one account and arrive in the other, together, as one indivisible action — never a state where it's left your account but hasn't yet reached the other, even if the power goes out mid-transfer.
What a transaction guarantees
terminalsql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;
-- both updates succeed together, or if anything fails, ROLLBACK undoes both
The four letters behind that guarantee
LetterGuarantees
AtomicityAll steps succeed together, or none of them apply at all
ConsistencyThe database moves from one valid state to another, never a broken in-between
IsolationConcurrent transactions don't interfere with each other's in-progress work
DurabilityOnce committed, it survives a crash immediately after
BEGIN / COMMIT / ROLLBACKACIDisolation levels

Normalization vs Denormalization
The trade-off underneath every schema design decision
▼
⚖️
Think of it likeStoring a recipe book by ingredient, cross-referenced by recipe number (normalized, no repeated typing) versus printing every recipe out in full every single time, ingredients and all (denormalized, faster to read, wastes paper, and a typo means fixing it in a hundred places).
Normal forms, briefly
FormRule
1NFEvery column holds one atomic value, no repeating groups
2NFEvery non-key column depends on the whole primary key, not part of it
3NFNo non-key column depends on another non-key column
Why real systems intentionally break these rules

A fully normalized schema minimizes duplication but requires more joins to answer a question. A denormalized schema duplicates some data on purpose, trading storage space and update complexity for read speed — exactly what analytics and reporting systems often do, since they read far more often than they write.

1NF / 2NF / 3NFdenormalizationread-heavy vs write-heavy design

05 · Corrections

What SQL is not

Myth

SQL is old technology being replaced by newer tools.

Reality

SQL is over 50 years old and is, if anything, more widely used today than ever — even most "NoSQL-first" companies eventually add a SQL layer on top for analytics.

Myth

You need to memorize dozens of join types.

Reality

There are really only three shapes — INNER, LEFT/RIGHT (mirrors of each other), and FULL OUTER — plus the NULL-check trick that turns any of them into an anti-join. Everything else is a variation on those.

Myth

A slow query means you need a bigger database server.

Reality

The overwhelming majority of slow queries in the real world are fixed by an index or a schema change (stages 18 & 20), not more hardware.

Myth

RIGHT JOIN is a completely different concept from LEFT JOIN.

Reality

They're mirror images of the exact same operation — most style guides recommend never even using RIGHT JOIN, and just swapping the table order in a LEFT JOIN instead, for consistency.

06 · Common questions

Frequently asked, honestly answered

Do I need to know how to code to learn SQL?
No — SQL is often the very first "code" people learn, precisely because it reads close to plain English. It's a genuinely reasonable starting point even with zero programming background.
Which database should I actually practice on?
PostgreSQL is a strong free default — it's close to the SQL standard, widely used professionally, and every query on this page runs on it with little to no modification.
Is SQL enough on its own to get a data job?
For many analyst roles, yes, combined with a visualization tool. For engineering roles, SQL is necessary but usually paired with a language like Python and the DevOps fundamentals from the earlier path on this site.
Why does the same query sometimes run fast and sometimes slow?
Usually the amount of data changed, an index got dropped or wasn't used, or the database's query planner made a different choice based on updated statistics (stage 22) — not because SQL itself is inconsistent.
What's the very first thing I should actually practice, hands-on?
Install PostgreSQL (or use a free online sandbox), create two small related tables by hand, and try writing an INNER JOIN, then a LEFT JOIN, then a LEFT JOIN with a NULL check between them. Those three queries alone build most of the intuition everything else depends on.
What happens if I never learn any of this?
Nothing, if you're not planning to work with data directly — every app you use already runs these queries invisibly on your behalf, the way the "Day in the Life" story above describes. This page exists so the words stop being a wall of jargon the next time you hear them.

How many rows on one side of a relationship match rows on the other — one-to-one, one-to-many, or many-to-many.
Alias
A temporary name given to a table or column within a query, using AS, to make results or self-joins readable.
Result set
The table of rows a query returns — itself temporary, and gone once the query finishes.
DDL / DML
Data Definition Language (structure: CREATE, ALTER) versus Data Manipulation Language (data: INSERT, UPDATE, DELETE).
Constraint
A rule enforced by the database itself, like PRIMARY KEY or NOT NULL, that rejects data violating it.
Full table scan
Checking every row in a table one by one, because no usable index exists for the query.
Idempotent
Running the same operation multiple times has the same effect as running it once — relevant when re-running migration scripts safely.

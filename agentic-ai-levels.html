A note on the format: this page is built to teach, not to impress — one level at a time, no autoplay, no narration racing ahead of the text. Each level ends with a short question, because being quizzed on something works far better for memory than just reading it again.

Seven levels, from a scripted chatbot to agents that delegate to other agents.

Each level adds exactly one real capability the previous one didn't have. Go in order the first time — the whole point is that level 4 only makes sense once you've felt the limits of level 3.

LEVEL 1 OF 7

Script Bots

No intelligence — scripted input mapped to scripted output.

Query
→
Scripts
→
Output
e.g. "if user says X, reply with Y" — a decision tree, not a model

Why this is the floor

A script bot has no model of language at all — it's pattern-matching against phrases someone anticipated in advance. Ask it something the author didn't foresee, and it fails outright or falls back to a generic default. There's no understanding to fall back on, because there was never any understanding to begin with.

The actual leap from level 1

An LLM isn't following an "if this, then that" rule someone wrote. The attention mechanism lets it weigh which words in a sentence matter most to each other, so it generalizes to phrasings it never saw verbatim — which is exactly what a script bot can't do.

But it's still a closed box: everything it "knows" was frozen at training time. It can't look anything up, and it forgets everything the moment the conversation ends.

LEVEL 3 OF 7

LLMs + Tools

The model can now call external functions through APIs — it can act, not just talk.

⇄

(via unique APIs)
→
Output
the model decides mid-response that it needs a tool, calls it, then continues

The actual leap from level 2

A plain LLM can only tell you what it remembers from training. Give it a calculator tool, and it can compute an exact answer instead of guessing. Give it a weather API, and it can tell you today's forecast — something no amount of training data could have covered, because "today" didn't exist when it was trained.

The catch: each tool needs its own custom integration, hand-built for that specific API. There's no standard way to plug in a new one yet, and the model still isn't grounding answers in a searchable body of your own documents.

LEVEL 4 OF 7

RAG — Retrieval-Augmented Generation

The agent dynamically pulls relevant data before answering, instead of answering from memory alone.

→

→

(relevant chunks found)
→
Augmented
(context + prompt)
→
Generation → Output
R-A-G: Retrieval finds evidence, Augmented context assembles it, Generation writes the answer

The actual leap from level 3

Tools alone give the model one-off actions. RAG gives it a searchable memory of your actual content — internal docs, a knowledge base, current web results — retrieved fresh for every query and inserted directly into the prompt before the model answers.

This is the mechanism, not a marketing claim, behind why RAG systems hallucinate less: the model is answering from evidence sitting right in its context window, not purely from what it memorized months or years earlier.

LEVEL 5 OF 7

Agentic Memory

The agent now remembers — both within a session and across sessions over time.

Query
⇄
Agent
⇄
Short: cache, working Long: semantic, episodic, procedural
short-term memory lasts a session; long-term memory persists and is deliberately categorized

The actual leap from level 4

RAG retrieves facts from a static knowledge base. Memory is different — it's the agent's own accumulated experience. Semantic memory holds general facts it's learned; episodic memory holds specific past events ("last week you asked about X and preferred Y"); procedural memory holds learned skills or workflows.

This is what makes an agent feel continuous across conversations instead of starting from zero every single time — a distinction that matters when you're evaluating whether a product's "memory" is real or just a longer prompt.

LEVEL 6 OF 7

MCP — Model Context Protocol

A standard way to connect an agent to enterprise tools, instead of a custom integration for every one.

Query
⇄
Agent
⇄
MCP Server
→
PlaywrightChroma…any server
one protocol, many servers — no bespoke integration code per tool

The actual leap from level 3's tools

Back at level 3, every tool needed its own hand-built API integration — real engineering work, repeated for each new tool. MCP replaces that with one shared protocol: any tool that speaks MCP can be plugged in without custom glue code.

This is the difference between "we integrated five tools, painstakingly, one at a time" and "we support any tool that implements this one open standard."

LEVEL 7 OF 7

A2A Agents — Agent to Agent

A main agent delegates parts of a task to specialized sub-agents and coordinates their work.

Query
⇄
Main Agent
(tools, db, cache)
⇄
A2A Protocol
⇄
Sub Agent
(tools, db, cache)
each agent can have its own tools and memory — this is composition, not just one bigger agent

The actual leap from level 6

MCP connects one agent to many tools. A2A connects many agents to each other. A main agent can break a complex task into pieces, hand each piece to a sub-agent specialized for it, and assemble the results — the same reason large software systems are built from many small services instead of one giant program.

This is the top of the ladder because it's compositional: a well-built sub-agent at level 7 is itself potentially everything from level 1 through 6, wrapped up and delegated to.

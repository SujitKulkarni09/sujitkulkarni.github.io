The Problem

The vending‑machine trap

Most people use AI like a vending machine — insert a prompt, take the output, walk away. Every session starts from zero. Every mistake gets made again next week. The fix isn't a smarter model. It's a smarter loop.

One‑off prompting

Single input prompt → single completion → accepted or discarded. No persistence, no retrieval, no audit trail. Errors are corrected in the moment and forgotten by the next session. You ask, it answers, you move on. Next week you hit the exact same problem — and fix it the exact same way, from scratch.

Compounding loop

Every run feeds a reference log. Every failure becomes a heuristic. Every heuristic gets retrieved next time. An independent auditor keeps the loop honest. You keep a short notebook of "what worked." You load it before every new task. Small rules stack up into real expertise — yours, not the model's.

Implementation notes
Storage
  • Use a simple database or git for logs; move to a vector DB (Pinecone, Milvus, Weaviate) once you need semantic search. Start with a folder or a git repo; upgrade only when you need smart search.
Retrieval
  • Prepend retrieved rules/examples as system messages or in-context examples; watch your token budget so nothing gets truncated. Load only the most relevant rules so you don't hit size limits.
Verification
  • Start with deterministic checks — required fields, regex for URLs, numeric thresholds. Add classifier checks or human review for anything subjective. Use basic tests first; bring in humans for judgement calls.
Safety
  • Watch for prompt injection and retrieval poisoning; validate retrieved rules and prefer signed or verified sources. Don't blindly trust retrieved rules — check where they came from.
Scaling
  • At high volume, automate distillation — turn repeated failures into new rules via a small script or verifier agent — and batch updates to the policy store. If you use this a lot, automate the rule-creation step too.
Prompt template — Weekly report generator
System instruction
You are a factual, concise report generator. Always follow the retrieved rules provided in <<RETRIEVED_RULES>>. If a rule conflicts with another, prioritize the most recent rule. Mark any information you cannot verify with "[UNVERIFIED]". After generating, output a JSON object with keys: "report_text", "sources" (list of URLs with retrieval dates), and "notes". Do not perform verification — return the draft only.
User prompt
Generate a 400–600 word weekly analytics summary for <<PROJECT_NAME>> covering highlights, key metrics (traffic, conversions, retention), and recommended next steps. Context: <<PROJECT_CONTEXT>>. Retrieved rules: <<RETRIEVED_RULES>>. Required: include data source URLs with retrieval dates, list any assumptions, and highlight any data gaps. Output only the JSON object specified.
Verifier prompt
Given the draft JSON from the generator, check the following deterministically: 1) "report_text" length between 400–600 words; 2) every item in "sources" is a valid URL and includes a retrieval date (YYYY-MM-DD); 3) "notes" lists assumptions or data gaps. Return a JSON verdict: {"pass": true/false, "fail_reasons": [...], "fix_suggestions": [...]}.
Runtime flow
  • Retrieve top N rules/examples into <<RETRIEVED_RULES>> via RAG.
  • Send system + user prompt to the model.
  • Save the generator's output to logs and the vector store.
  • Run the verifier — publish on pass; on fail, save the fix as a new rule and queue a regeneration.
Prompt template — Content editor
System instruction
You are a copy editor focused on clarity, tone, and factual accuracy. Follow retrieved style rules in <<STYLE_RULES>>. Return two outputs: "edited_text" (the revised content) and "edit_log" (list of changes and reasons). Don't self-approve — only edit.
User prompt
Edit the following article for tone (professional, 3rd-person), clarity, and accuracy. Preserve facts. Context: <<ARTICLE_CONTEXT>>. Style rules: <<STYLE_RULES>>. Required: produce "edited_text" and "edit_log" with each change (location, original phrase, new phrase, rationale).
QA verifier prompt
Validate the editor output: 1) tone matches target (professional, 3rd-person); 2) no fact contradictions with <<FACT_BASE>> (list provided); 3) edit_log explains each substantive change. Return {"pass": true/false, "issues": [...], "corrections": [...]}.
Implementation tips
  • Replace every <<...>> placeholder at runtime with retrieved content, context, or the latest rules.
  • Keep the verifier separate and deterministic where possible — regex checks, length, URL formats — and fall back to human QA only for subjective items.
  • Log every input, output, retrieved rule, and verifier result to your episodic store so future runs can retrieve it.

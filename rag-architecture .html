Stage 01 / Preparation

Ingestion — getting content ready

Before anything can be retrieved, it has to be parsed, split, converted to vectors, and stored — the unglamorous stage that determines the ceiling for everything downstream.

Document Parsing

Extracts clean text from source files — PDFs, Word docs, HTML — stripping layout noise so downstream steps work with usable content. Poor parsing caps quality no later step can fully recover from.

Chunking

Splits long documents into smaller, retrievable pieces. Chunk size is a real tradeoff: too small loses context, too large drags in irrelevant text and wastes context window space later.

Embeddings

Converts each chunk into a numeric vector that captures its semantic meaning, so chunks about similar ideas land close together in vector space — even if they don't share exact words.

Ingestion Pipeline

The automated sequence tying parsing, chunking, and embedding together, continuously feeding new or updated content into the retrieval system rather than a one-time load.

Vector Database

Stores embedding vectors efficiently and supports fast nearest-neighbor search — the index that makes searching millions of chunks by meaning practical at low latency.

Knowledge Base

The underlying trusted content collection — internal documentation, a product catalog, external sources — that the whole pipeline exists to search accurately.

Stage 02 / Search

Retrieval — finding the right pieces

This is where a query actually becomes a shortlist of evidence — and where most of RAG's real engineering effort tends to go, because bad retrieval can't be fixed by a better prompt.

Query Expansion

Rewrites or broadens a user's query before searching, improving recall by capturing phrasings and synonyms the original query didn't use.

Retriever

The component that takes a query and finds the most relevant chunks, using vector similarity, keyword matching, or both — the first real search stage of RAG.

Hybrid Search

Combines lexical (keyword) and semantic (vector) search. Keyword search catches exact terms and codes that embeddings can blur; vector search catches meaning keywords miss.

Similarity Score

A numeric measure of how close a chunk's embedding is to the query's embedding, used to rank candidates before anything gets shown to the model.

Top-K Retrieval

Controls how many of the highest-scoring chunks get passed forward. Too few misses relevant context; too high adds noise and burns context-window budget.

Metadata Filtering

Narrows results using structured attributes — date, source, tag, department — before or alongside semantic search, when "relevant" also has to mean "current" or "authorized."

Re-Ranking

A second pass that reorders the initially retrieved chunks with a more precise, often slower model — trading extra latency for meaningfully better final relevance.

Stage 03 / Answering

Generation — assembling the answer

Retrieval found the evidence; this stage decides how much of it fits, how it's presented to the model, and how tightly the final answer has to stay tied to it.

Context Window

The maximum amount of text the model can process at once — which directly caps how many retrieved chunks can actually be used, no matter how good retrieval was.

Tokenization

Breaks text into the model's basic processing units. Directly affects cost and context limits — chunk sizing and prompt design both have to work in tokens, not characters.

Prompt Template

The structured format that combines the user's task with the retrieved evidence into one consistent instruction — improving consistency across thousands of varied queries.

Grounding

Anchors the model's output to the retrieved source material. This is the actual mechanism — not a vague promise — that makes RAG more trustworthy than a model answering from memory alone.

Semantic Cache

Stores responses to previously seen — or semantically similar — queries, cutting cost and latency on repeated or near-duplicate requests without re-running retrieval and generation.

Stage 04 / Trust

Quality & operations — staying good

Not a final checklist — a set of measurements and safeguards that run continuously, the same way governance wraps every layer rather than gating it once at the end.

Latency

The time to return a response — a combination of retrieval time plus generation time, and usually the single most user-visible metric in the whole system.

Throughput

How many requests the system handles over a given period. Where latency is felt by one user, throughput is what determines whether the system survives real traffic at scale.

Hallucination Mitigation

The combined set of techniques — grounding, source checks, evaluation — used to reduce fabricated or unsupported answers. No single technique here is sufficient alone.

Source Attribution

Links a generated answer back to the specific chunks it drew from, improving transparency and letting a user actually verify a claim instead of trusting it blindly.

Evaluation

Systematic testing of RAG quality — measuring relevance, faithfulness to sources, and answer correctness — to guide tuning decisions instead of relying on spot-checks.

Guardrails

Safety and policy checks applied to both input and output, controlling risky behavior and protecting users and data — the same continuous-not-final principle as model governance.

Feedback Loop

Captures user feedback and downstream signals to continuously improve prompts, retrieval quality, and ranking — turning production usage into the next round of tuning data.

STAGE 01
Ingestion runs continuously
Not a one-time load — new or updated content keeps flowing through parsing, chunking, and embedding as the knowledge base changes.
STAGE 02
Retrieval is where most tuning happens
Query expansion, hybrid search, and re-ranking exist because a single-pass semantic search alone reliably misses relevant results.
STAGE 03
Generation is bounded by context
However good retrieval is, the context window and tokenization limits decide how much of that evidence the model actually sees.
STAGE 04
Quality is a loop, not a gate
Evaluation and the feedback loop feed back into retrieval and prompt design — the pipeline is meant to keep improving, not just ship once.
